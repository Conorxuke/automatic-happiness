Implementation Thinking · for the Adelin conversation

Four engines. One dashboard.

The back office isn't five bespoke agents — it's four reusable engines, configured as “toppings,” with one control surface a few people run. The same four engines we'd build for Medtronic.

The point to land with Adelin

Build the four bases once; configure them endlessly. A new back-office need isn't a new system — it's a new topping on an engine that already exists. And it's the same architecture as the Medtronic DHF programme: Extract, Draft, Audit, Retrieve. So this build isn't throwaway internal plumbing — it's the first working instance of the product Trinzo wants to sell. Trinzo becomes the first user of its own engine.

What the dashboard gives a few back-office people

  • One place to work — they never touch Supabase, Airia, or a model. They approve, edit, trigger, and read reports from a single screen.
  • The 80/10/10 made operational — every engine output lands in a queue; nothing leaves without a human yes. The dashboard is the human-in-the-loop.
  • Reporting as a byproduct — RAG health, SLA flags, R&D evidence log and token spend are views, not extra work.

What this changes about the build

  • 4 things to build & maintain, not 5–10 bespoke agents. Cheaper to run, far cheaper to maintain.
  • Config, not code, for each new function — closer to something the coordinator can extend later.
  • Provenance on every output (engine, model version, timestamp) — the discipline that makes the same engines safe for client/GxP work.
